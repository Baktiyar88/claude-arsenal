## Контекст

Нужен публичный репозиторий, который собирает в одном месте все инструменты/репо/промпты/хуки/методологии, прокачивающие Claude Code. Потенциально в него будут контрибьютить разные люди (друзья, коллеги), поэтому структура должна:

- быть понятной коллеге, который **детально не изучал** содержимое ссылок
- масштабироваться на десятки и сотни ресурсов
- позволять за 15 секунд понять "надо мне это или нет" по одной карточке
- соответствовать нормам публичных awesome-list (legal, с атрибуцией авторов)

Примеры-эталоны, которые легли в основу категоризации:
- [obra/superpowers](https://github.com/obra/superpowers) — полный фреймворк методологии (skills + agents + commands + hooks)
- [safishamsi/graphify](https://github.com/safishamsi/graphify) — инструмент + MCP-сервер, превращающий код в knowledge graph

## Цели

- **G1** — Коллега открывает README и через 10 секунд понимает, какая папка ему нужна, опираясь на свою задачу ("что тебе нужно?").
- **G2** — Добавление нового ресурса занимает < 1 минуты (PR или issue по шаблону).
- **G3** — Каждый ресурс описывается унифицированной карточкой: что даёт / когда брать / для чего лучше / установка / автор.
- **G4** — Структура не требует пересмотра при росте от 5 до 100+ ресурсов.
- **G5** — Соблюдение лицензий и авторства — только публичные репо с открытой лицензией.

## Non-goals (что НЕ делаем в этой итерации)

- **N1** — Не копируем код чужих репо внутрь нашего (только ссылки + описания; это вариант A из брейншторма).
- **N2** — Не подключаем git submodules (вариант B отклонён).
- **N3** — Не извлекаем артефакты из чужих репо в наши папки (вариант C отклонён).
- **N4** — Не делаем автоматизацию установки ресурсов (никаких install-скриптов на первой итерации).
- **N5** — Не делаем CI/линтер для README (ручная проверка на старте).

## Дизайн

### 1. Структура папок

```
claude-arsenal/
├── README.md                      # Главный гид "что тебе нужно?"
├── CONTRIBUTING.md                # Инструкция по добавлению ресурса
├── LICENSE                        # MIT
│
├── frameworks/                    # Целостные методологии "всё-в-одном"
│   └── README.md                  # пример: superpowers
├── orchestrators/                 # Многоагентные системы / оркестраторы
│   └── README.md                  # пример: твой оркестратор для Python бэка
├── agents-skills/                 # Отдельные агенты и скиллы
│   └── README.md                  # пример: planner, reviewer, architect
├── tools/                         # MCP-серверы, инструменты, утилиты
│   └── README.md                  # пример: graphify
├── automation/                    # Хуки settings.json, промпты, шаблоны CLAUDE.md
│   └── README.md
├── docs/                          # Гайды, шпаргалки, сравнения
│   ├── README.md
│   └── superpowers/specs/         # мета-спеки этого самого репо (как эта)
│
└── .github/
    └── ISSUE_TEMPLATE/
        └── add-resource.yml       # Форма "предложить ресурс"
```

**Логика папок (гайд для навигации):**

| Папка | Что туда | Как отличить от соседей |
|---|---|---|
| `frameworks/` | Целостные методологии с несколькими компонентами сразу (skills + hooks + commands) | "Всё-в-одном пакет" |
| `orchestrators/` | Системы, управляющие несколькими агентами одновременно | "Дирижирует другими агентами" |
| `agents-skills/` | Отдельные, самодостаточные агенты или скиллы | "Один инструмент, одна задача" |
| `tools/` | MCP-серверы, CLI-утилиты, внешние инструменты | "Требует установки как отдельный сервис/пакет" |
| `automation/` | Хуки settings.json, PostToolUse/PreToolUse, CLAUDE.md шаблоны, промпт-шаблоны | "Конфиги и шаблоны, не код" |
| `docs/` | Написанное пользователем: гайды, шпаргалки, сравнения | "Текст/документы, не исполняемое" |

### 2. Формат карточки ресурса

Карточка в README подпапки:

```markdown
### [название-репо](https://github.com/user/repo)

**Что даёт:** одно предложение про полезность
**Когда брать:** одно предложение про use-case
**Для чего лучше:** универсал / backend / frontend / mobile / data
**Установка:** команда клона или ссылка на инструкцию
**Автор:** [@username](https://github.com/username)
```

**Обязательные поля:** все пять.
**Принципы:**
- "Что даёт" и "Когда брать" — максимум одно предложение каждое, чтобы коллега не утонул в описаниях.
- "Для чего лучше" — одно или несколько значений через запятую из закрытого списка: `универсал`, `backend`, `frontend`, `mobile`, `data`.
- Название ресурса — это и есть ссылка на оригинальный репо.
- Атрибуция автора обязательна (уважение авторства в awesome-list-нормах).

### 3. Главный README — стиль "что тебе нужно?"

Структура корневого `README.md`:

1. **Шапка и пояснение** (2-3 строки) — что такое этот репо.
2. **Блок "🎯 Что тебе нужно?"** — задачи от лица пользователя с прямыми ссылками в карточки:
   - "Хочу универсал с хорошей памятью и контекстом" → [superpowers]
   - "Хочу оркестратор для Python бэкенда" → [твой-оркестратор]
   - "Хочу превратить кодовую базу в граф знаний" → [graphify]
   - "Хочу готовые хуки и шаблоны CLAUDE.md" → [automation/]
   - (этот список растёт с добавлением ресурсов)
3. **Таблица "📁 Все папки"** — компактный обзор шести папок с ссылками и парой примеров.
4. **Блок "🤝 Добавить своё"** — ссылка на CONTRIBUTING и на issue template.

Главная ценность: коллега читает **задачами**, а не терминами. Он не обязан понимать что такое "MCP-сервер" или "оркестратор" — он идёт по своей нужде.

### 4. CONTRIBUTING.md

Четыре пункта, каждый короткий:

1. **Найди нужную папку** — таблица с шестью папками и описанием.
2. **Открой README папки и добавь карточку в конец** — готовый шаблон для копипаста.
3. **Правила** — только публичные репо с открытой лицензией; ресурс должен работать; не дублировать.
4. **Сделай PR или заведи issue по шаблону** — два пути для разного уровня комфорта с git.

### 5. Issue template (`.github/ISSUE_TEMPLATE/add-resource.yml`)

Форма в GitHub UI с полями:
- Ссылка на репо (обязательное)
- Категория (dropdown: frameworks / orchestrators / agents-skills / tools / automation / docs)
- Что даёт (textarea, обязательное)
- Когда брать (textarea, обязательное)
- Для чего лучше (dropdown: универсал / backend / frontend / mobile / data)

Заполняется за 30 секунд. Владелец репо переносит в README подпапки одним копипастом.

### 6. Лицензия и правовой режим

- **Лицензия этого репо:** MIT.
- **Подход:** awesome-list, только ссылки + описания. Юридически эквивалентно публикации bookmarks.
- **В CONTRIBUTING явно написано:** "Только публичные репо с открытой лицензией (MIT, Apache, GPL и т.п.). Атрибуция автора обязательна."
- **Если в будущем захотим копировать код внутрь** — отдельная итерация с анализом лицензий каждого ресурса.

## План имплементации (верхнеуровнево)

Будет детализирован в отдельном implementation plan через skill `writing-plans`. Верхнеуровневые шаги:

1. Создать 6 папок + пустые README с заголовками и пустыми списками.
2. Написать корневой `README.md` со стилем "что тебе нужно?" и заполнить первыми 2 примерами (superpowers, graphify) + оркестратор пользователя если даст ссылку.
3. Написать `CONTRIBUTING.md`.
4. Написать `LICENSE` (MIT).
5. Создать `.github/ISSUE_TEMPLATE/add-resource.yml`.
6. Первый коммит, пуш на `origin/main`.
